## سجل القرارات المعمارية لمشروع ile-key-v3

هذه الوثيقة تسجل القرارات التقنية والاستراتيجية المهمة التي توجه تطوير `ile-key-v3` من نسخة ERPNext v11 قديمة إلى منصة رقمية احترافية لإدارة الصادرات.

الهدف من هذه الوثيقة هو منع التشتت التقني، والحفاظ على القرارات المهمة داخل GitHub حتى لو تم مسح البيئة المحلية بسبب Deep Freeze.

## ADR-000: اعتماد وثائق GitHub كمصدر استمرارية رسمي

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** الاستمرارية والحوكمة

### السياق

جهاز التطوير يعمل على Windows 11 مع Deep Freeze، ما يعني أن البيئة المحلية وبياناتها غير مضمونة بعد إعادة التشغيل أو انقطاع الكهرباء.

### القرار

اعتبار GitHub هو المصدر الرسمي للكود والوثائق، واعتبار backups الخارجية هي المصدر الرسمي للبيانات.

### العواقب

- يجب commit و push أي تقدم مهم.
- لا يجوز الاعتماد على ملفات محلية غير محفوظة.
- يجب تحديث الوثائق مع أي قرار معماري مهم.

## ADR-001: عدم الاعتماد على Windows local كمصدر وحيد

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** البنية التحتية

### السياق

Frappe/ERPNext يعملان بشكل أنظف داخل Linux/WSL/Docker. تشغيلهما محلياً على Windows ممكن للتجربة، لكنه غير مناسب كمصدر وحيد للتطوير أو الإنتاج.

### القرار

استخدام Windows local فقط كواجهة تشغيل مؤقتة، مع بناء بيئة تطوير قابلة لإعادة الإنشاء داخل:

```text
WSL2 / Ubuntu
Docker Compose
frappe_docker
Cloud Dev Server
```

### العواقب

- يجب توثيق خطوات الاستعادة.
- يجب اختبار backup/restore.
- يجب عدم تخزين بيانات مهمة فقط داخل `C:`.

## ADR-002: عدم تعديل ERPNext core مباشرة في النظام النهائي

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** بنية التطبيق

### السياق

المشروع الحالي مبني على ERPNext v11، وقد تحتوي النسخة الحالية على تعديلات داخل تطبيق ERPNext نفسه. هذا النمط يجعل الترقية صعبة ويزيد خطر التعارضات.

### القرار

في النظام النهائي يجب نقل كل تخصيصات الأعمال إلى custom app مستقل، مقترح باسم:

```text
nile_export
```

ويُترك ERPNext/Frappe core قدر الإمكان غير معدل.

### العواقب

- سهولة الترقية.
- فصل واضح بين core و business logic.
- تقليل مخاطر فقدان التعديلات.
- الحاجة إلى جهد نقل من `erpnext/` إلى `nile_export/`.

## ADR-003: التوجه إلى Frappe/ERPNext حديث للنظام النهائي

**الحالة:** مقبول مبدئياً  
**التاريخ:** 2026-06-22  
**المجال:** استراتيجية الترقية

### السياق

الإصدار الحالي المكتشف هو ERPNext `11.1.49`، وهو إصدار قديم مقارنة بالإصدارات الحديثة المدعومة.

### القرار

التوجه الاستراتيجي إلى Frappe/ERPNext v16 أو أحدث إصدار مستقر مدعوم عند بدء البناء النهائي، مع اعتبار v15 خياراً مؤقتاً أقل مخاطرة إذا تطلب العمل ذلك.

### العواقب

- تحسين الأمان والدعم.
- توافق أفضل مع Python/Node الحديث.
- حاجة إلى خطة ترقية أو إعادة بناء.
- ضرورة اختبار migrations بدقة.

## ADR-004: اعتماد custom app للوظائف الخاصة بالتصدير

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** نموذج الأعمال

### السياق

منصة Nile Key ليست مجرد ERP داخلي، بل بوابة رقمية لإدارة التصدير والامتثال والمستندات والتكاملات.

### القرار

إنشاء custom app مستقل يحتوي على DocTypes و Workflows و APIs الخاصة بالتصدير، مثل:

```text
Export Shipment
Shipment Document
ACID Record
Customs Declaration
Supplier Station
Importer
RFQ
Export Quotation
Compliance Check
Logistics Provider
Government Integration Log
```

### العواقب

- وضوح معماري.
- قابلية تطوير أعلى.
- سهولة اختبار وحدات التصدير.
- فصل بيانات الأعمال عن ERP core.

## ADR-005: استخدام بوابة Next.js منفصلة للواجهة العامة

**الحالة:** مقبول مبدئياً  
**التاريخ:** 2026-06-22  
**المجال:** الواجهة الرقمية

### السياق

الصفحة الرئيسية للشركة يجب أن تكون مستقلة عن واجهة ERPNext login، وتعكس هوية مؤسسية راقية.

### القرار

استخدام Next.js كبوابة عامة/خاصة اختيارية، مع ربطها بـ Frappe APIs بشكل آمن.

### العواقب

- واجهة أسرع وأكثر مرونة.
- فصل تجربة المستخدم العامة عن Desk.
- ضرورة إدارة المصادقة والـ API tokens بحذر.
- عدم وضع أسرار backend داخل المتصفح.

## ADR-006: الصفحة الرئيسية لا تكون صفحة login افتراضية

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:**routing / UX

### السياق

في Frappe v11 قد تكون الصفحة الافتراضية `/login` إذا لم تُضبط `Website Settings.home_page`.

### القرار

تعيين صفحة مستقلة كـ homepage، أو توجيه `/` إلى بوابة مستقلة، مع إبقاء `/login` و `/desk` متاحين للوصول الإداري.

### العواقب

- تجربة دخول احترافية.
- فصل الواجهة التسويقية عن لوحة ERP.
- ضرورة تحديث cache بعد تغيير hooks أو Website Settings.

## ADR-007: اعتماد RBAC صارم على مستوى الخادم

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** الأمان

### السياق

النظام سيتعامل مع موردين وعملاء وشحنات ومستندات تصدير حساسة.

### القرار

تطبيق صلاحيات على مستوى DocType و API و Workflow، وليس فقط عبر إخفاء عناصر الواجهة.

### العواقب

- حماية أفضل للبيانات.
- تعقيد أعلى في تصميم الصلاحيات.
- ضرورة اختبار permissions لكل دور.

## ADR-008: اعتماد audit logs للعمليات الحساسة

- زيادة حجم البيانات المسجلة.

## ADR-009: عدم تخزين الأسرار داخل Git

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** الأمان

### السياق

المشروع قد يحتوي أو يحتاج إلى مفاتيح API وبيانات اتصال وcredentials.

### القرار

يمنع تخزين الآتي داخل Git:

```text
.env
site_config.json الحقيقي
كلمات مرور
API keys
SMTP credentials
Payment credentials
Supabase service keys
```

### العواقب

- أمان أعلى.
- ضرورة استخدام secrets manager أو ملفات محلية مشفرة.
- ضرورة `.gitignore` محكم.

## ADR-010: اعتماد backups مشفرة وخارجية

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** الاستمرارية

### السياق

Deep Freeze قد يمسح قاعدة البيانات والملفات المحلية.

### القرار

تنفيذ backup دوري لـ:

```text
database
public files
private files
site_config غير الحساس أو مشفر
fixtures
```

وحفظها خارجياً بشكل مشفر.

### العواقب

- تقليل خطر فقدان البيانات.
- ضرورة اختبار restore.
- عدم رفع backups حساسة إلى مستودع عام.

**الحالة:** مقبول مبدئياً  
**التاريخ:** 2026-06-22  
**المجال:** الجودة والنشر

## ADR-012: اعتماد staging قبل production

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** النشر

### السياق

النظام يستهدف عمليات تصدير حساسة وقد يتكامل مع جهات خارجية.

### القرار

لا يتم النشر إلى production مباشرة. يجب المرور بـ:

```text
development
staging
UAT
production
```

### العواقب

- تقليل مخاطر النشر.
- زيادة وقت دورة الإطلاق.
- حاجة إلى خطة rollback.

## ADR-016: عدم اعتبار البيئة المحلية الحالية بيئة إنتاجية

**الحالة:** مقبول  
**التاريخ:** 2026-06-22  
**المجال:** التشغيل

### السياق

تم تثبيت MariaDB و Redis محلياً على Windows للتجربة، لكن الجهاز يستخدم Deep Freeze.

### القرار

اعتبار البيئة المحلية الحالية disposable فقط، وليست أساساً للإنتاج أو حتى للتطوير طويل المدى.

### العواقب

- ضرورة بناء بيئة مستقرة.
- ضرورة backups.
- عدم الوثوق بالبيانات المحلية.
